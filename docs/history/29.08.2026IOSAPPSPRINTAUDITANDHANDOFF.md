## 1. Executive Summary

RemoteCTRL is a well-structured monorepo containing an Express + SSH backend and an Expo SDK 52 React Native mobile client. The app provides remote SSH control, file browser, process manager, and push notifications.  

**Critical Issue:** iOS TestFlight builds (v1.0.2, v1.0.3, v1.0.4) crash on cold start with `EXC_CRASH (SIGABRT)` from an uncaught `NSException` in native code. The crash is isolated to a native module during early app launch before JavaScript execution. `expo-notifications` was removed in v1.0.4; build is currently in flight for testing.  

**Overall Quality:** High. Clean separation of concerns, typed APIs via Zod/OpenAPI, proper credential redaction, and good error handling. Mobile bootstrap correctly defers side effects; server uses connection pooling and async WebSocket relay.  

**Key Risks:**
- Native crash blocks iOS release.
- ASC API key material stored in absolute Windows path (`eas.json:47`).
- EAS quota exhausted; project migrated to new Expo account.
- Windows service installer was recently fixed to target `RemoteCTRL.exe` correctly.

## 3. Mobile Application Audit

### 3.1 App Bootstrap & Crash Mitigation

`artifacts/mobile/app/_layout.tsx` is the root component and has been hardened:

- All potentially risky side effects are deferred to `useEffect` after font load.
- `bootstrapBackground()` wraps:
  - `SplashScreen.preventAutoHideAsync()` (with error logging)
  - `AsyncStorage` overrides for `server-url` and `api-token`
- Global error traps installed in module top (wrapped in try/catch).
- `ErrorBoundary` wraps the entire stack to catch render errors.

This design is correct for avoiding native module crashes on cold start.  

**Observations:**
- `expo-notifications` dependency removed in `package.json` (v1.0.4) to isolate crash.
- `setBaseUrl` and `setApiToken` are called at module top, but these are pure JS and safe.
- Debug logger sends logs to a developer rig IP; ensure it is stripped or disabled in production builds.

### 3.2 Screens & Navigation

- `expo-router` with stack + tab navigators.
- Onboarding wizard (3 steps) persisted via `AsyncStorage`.
- Terminal screen (`session/[sessionId].tsx`) uses WebSocket directly (bypasses API client) for SSH I/O. ANSI color rendering implemented; output ring buffer limited to 5000 lines.
- Connection screen (`connection.tsx`) manages multi-profile CRUD with password/SSH key auth and connection test.
- Tab screens: terminal (session list), files, processes, commands, settings.

All screens use generated React Query hooks (`@remotectrl/api-client-react`) for REST operations.

### 3.3 State Management & API Client

- API client (`lib/api-client-react/src/client.ts`) stores base URL and bearer token in `globalThis`. Simple, effective.
- Hooks use TanStack Query v5 with cache invalidation on mutations.
- Error handling: mutations surface errors; queries have retry=1, no refetch on window focus (good for mobile data usage).

Potential Enhancement: Provide a way to flush/clear React Query cache on logout.

### 3.4 Notifications

`lib/notifications.ts` is currently stubbed. Push registration and handling code would normally live here and in the Settings tab. The backend push route exists (`/api/push`) but is not used until re-enabled.

### 3.5 Debug Logging

`lib/debug-logger.ts` posts JSON logs to a developer rig (`10.0.0.127:8787` by default, configurable via `EXPO_DEV_LOG_HOST` or `RIG_IP`). It tries multiple candidates if the primary fails. This is invaluable for TestFlight debugging but should be gated by `__DEV__` or a build-time flag in production.

## 4. Backend API Server Audit

### 4.1 Core Architecture

- `src/index.ts` starts HTTP server, installs WebSocket handler, optionally starts Cloudflare Tunnel.
- `src/app.ts` configures middleware: CORS, JSON body (1 MB limit), pino-http logger, rate limiting, auth, then routes.
- `src/lib/sshManager.ts` is the SSH hub: single pooled `utilityClient` for exec/SFTP, per-session shell streams via `ssh2`. Output buffering (64 KB) during WS disconnection. Auto-reconnect logic for sessions (exponential backoff up to 10 attempts).
- `src/lib/wsHandler.ts` bridges WS messages to SSH sessions, handles resize, and sends output to client. Heartbeat via ping/pong every 30 s.
- `src/lib/store.ts` provides file-backed persistence (`data/store.json`) for connections, commands, push devices, and preferences. Synchronous `writeFileSync` is acceptable given infrequent writes.
- `src/lib/auth.ts` enforces Bearer token on all `/api/*` routes. `/health` is public.
- `src/lib/config.ts` reads `data/config.json` for `PORT`, `API_TOKEN`, `CLOUDFLARE_TUNNEL`.
- `src/lib/pushNotifications.ts` uses `expo-server-sdk` to send pushes; respects user preferences.
- `src/lib/tunnel.ts` manages Cloudflare Tunnel lifecycle.

### 4.2 Routes (21 endpoints per OpenAPI)

Key endpoints:
- Connection & profiles: `/api/connection`, `/api/connections`, `/api/connections/:id/activate`, `/api/connections/active`, `/api/connection/test` (rate-limited to 10/15min).
- Sessions: `/api/sessions` (create/list), `/api/sessions/:id` (close/rename).
- Files: full SFTP suite with path sanitization (`..` blocked), size limits (read ≤ 100 KB, upload ≤ 100 MB), and safe deletion (tries `unlink` then `rmdir`).
- Processes: `/api/processes` returns list; `/api/processes/:pid` kills after `/^\d+$/` validation.
- Commands: CRUD for saved command library.
- Push: device registration, preferences, device list, unregister.

All sensitive fields are masked in API responses via `store.ts` safe getters.

### 4.3 SecurityAudit

**Strong Points:**
- Bearer token auth on all API routes (configurable; dev mode when unset).
- Path traversal prevention (`sanitizePath` checks `..`).
- PID validation (`/^\d+$/`) before `Stop-Process`.
- Password/privateKey/passphrase redaction in logs and API responses (`pino` redact and `get*Safe` exports).
- Rate limiting: general (100/15min) and stricter on test endpoint (10/15min).
- Body size limit: JSON 1 MB; file upload 100 MB (multer memory storage; consider streaming to disk for large files).

**Concerns:**
- Absolute path for ASC API key in `eas.json:47` (`D:\\AgentDevWork\\repos\\AuthKey_RY256KG775.p8`). This should be a relative path or environment variable to avoid credential leakage.
- No CSRF protection because API is intended for mobile app with bearer token; acceptable.
- No account lockout on SSH password brute force; rate limiting at network layer only (might be fine for local use).
- `CLOUDFLARE_TUNNEL` defaults to `true` in `config.ts` and `eas.json`; ensure tunnel URL is rotated periodically or authenticated with Cloudflare.

### 4.4 Observability & Reliability

- pino-http logs every request with method, URL, status, duration.
- Global Express error handler returns `{ error, code }` JSON.
- `/health` returns `{ status, activeSessions, connectionConfigured, uptimeSeconds }` — useful for Docker health checks.
- Server shuts down gracefully on SIGTERM/SIGINT: closes SSH sessions and WebSocket server.

## 6. iOS Cold-Start Crash Investigation

### 6.1 Symptom

TestFlight builds (v1.0.2, v1.0.3, v1.0.4) crash immediately on launch with `EXC_CRASH (SIGABRT)`. Crash logs show uncaught `NSException` on a dispatch worker. Symbolicated offsets:
- v1.0.2: `0x2C65B8`
- v1.0.3: `0x2CADB8` (unchanged relative offset despite rebuild)

This indicates the same native code path faulting across two builds — strongly suggests an `expo-modules` autoregistration issue during `AppDelegate` startup, before JS bridge is ready.

### 6.2 Mitigation Taken (v1.0.4)

- Removed `expo-notifications` entirely from `package.json` and config to isolate suspect module.
- Deferred all native-touching side effects (splash screen, AsyncStorage) to `useEffect` after fonts load (already correct).
- Added robust debug logger to capture pre-crash logs, but logger itself may be too early; it's invoked at module top in `_layout.tsx`, albeit inside try/catch.

**Status:** v1.0.4 build `229f76c6` submitted to TestFlight; awaiting processing and install. If it still crashes, next suspects: `expo-secure-store` or `expo-local-authentication` (both have native components). Binary search strategy: remove one at a time until crash disappears.

### 6.3 Long-Term Fix Plan

Once the offending module is identified:
1. Add `@try/@catch` around module registration in `AppDelegate` via a config plugin or bare workflow.
2. Consider delaying non-critical native module initialization until after JS context loads (if Expo allows).
3. Re-enable push notifications with a defensive wrapper.

**Note:** dSYM symbolication on Windows is not available. For future certainty, consider macOS host for dSYM processing or use EAS's built-in symbolication.

## 9. Performance Considerations

- SSH connection pooling (utilityClient) avoids repeated handshakes for SFTP/exec.
- WebSocket message buffering (64 KB) handling temporary disconnects.
- Terminal output line buffer limited to 5000 lines.
- React Query cache with 10s stale time for file listings (good).
- No long-running JS timers on main thread beyond necessary (heartbeat is server-side).
